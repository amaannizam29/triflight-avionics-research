# Flight Controller Concept — Architecture

A block-level architecture of a flight controller (FC), at the level of "how it
is organized," not a full circuit schematic. It shows the microcontroller, the
sensors on their correct buses, the serial connectors, the motor outputs, and the
power stage.

**Scope note:** this is an architecture diagram, not a board-level schematic. It
documents how an FC is structured and why each interface is chosen. Board-level
schematic capture and PCB layout are a separate, deeper electronics task not
claimed here.

---

## The diagram in words

```
        IMU ─SPI─┐                      ┌─UART─ GPS
   Barometer ─I2C─┤      STM32F405      ├─UART─ Telemetry
                  │     (runs ArduPilot) ├─UART─ RC receiver
 Power 5V→3.3V ───┘                      └─timers─ ESC outputs
```

The STM32 is the brain. Everything else either feeds it data, receives commands
from it, or powers it.

---

## Why each block is there

### STM32F405 (the MCU)
The microcontroller that runs the autopilot firmware (ArduPilot). It reads the
sensors, runs the EKF and control loops, and drives the motor outputs. An F405 is
a common choice for this class of FC: enough processing for the control loop, and
widely supported by ArduPilot.

### IMU on SPI
The inertial measurement unit (gyroscope + accelerometer) provides angular rate
and acceleration, the fastest-changing, most timing-critical data on the board.
It sits on **SPI** because SPI is fast and low-latency, which matters for the
high-rate attitude loop. A laggy IMU bus would degrade control.

### Barometer on I2C
Measures altitude from air pressure. It updates slowly compared to the IMU, so it
does not need SPI's speed. **I2C** is fine and uses fewer pins. Matching the bus
to the data rate is the design point here.

### GPS, telemetry, RC on UART
These are serial links to/from other modules:
- **GPS** sends position over UART (often MAVLink or a GPS protocol).
- **Telemetry** radio carries MAVLink to the ground station, bidirectional.
- **RC receiver** carries the pilot's control input (UART/SBUS).
Each gets its own UART so they don't contend.

### ESC outputs on timers
The FC commands each motor's throttle through ESCs using **PWM or DShot** signals
generated by the MCU's hardware timers. On a QuadPlane that is five outputs (four
lift + one pusher).

### Power input (5V → 3.3V regulator)
The board takes input power (commonly 5V from the power module's BEC) and a
regulator steps it down to the **3.3V** the STM32 and sensors run on. In a full
schematic this stage also includes filtering and decoupling capacitors on every
chip's power pins; those are circuit-level details below the scope of this
architecture view.

---

## The one idea that ties it together

The recurring design decision is **matching each peripheral to the right bus**:

- SPI for the IMU because it is fast and timing-critical.
- I2C for the barometer because it is slow and pin-efficient.
- UART for serial modules (GPS, telemetry, RC) because they are serial streams.
- Timers for ESC outputs because motor signals are timed pulse trains.

Understanding *why* a sensor sits on a given bus is the electronics-architecture
knowledge this diagram demonstrates.
